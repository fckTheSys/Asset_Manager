# Cinema 4D 2026 Plugin Cheat Sheet

Практическая шпаргалка по разработке плагинов для Cinema 4D 2026 на Python. Документ собран как универсальный reference для следующих плагинов, а не как описание только одного проекта.

Основной фокус:
- `User Data`
- UI в `Attribute Manager` и `GeDialog`
- безопасная работа с `BaseContainer`, `DescID`, сообщениями и Python API
- типовые архитектурные паттерны
- реальные ограничения и поломки, на которые легко наткнуться в Cinema 4D 2026

## 1. Быстрая карта решений

Используй `User Data`, когда параметр:
- должен жить на объекте сцены
- должен анимироваться
- должен быть виден в `Attribute Manager`
- должен сохраняться прямо в сцене без отдельного окна

Используй `GeDialog`, когда действие:
- является командой, а не состоянием
- должно стабильно срабатывать по кнопке
- связано с созданием объектов, reset, bake, break links, import/export
- не обязано быть анимируемым параметром сцены

Используй `Python Tag`, когда нужна:
- логика, исполняемая на каждом кадре
- синхронизация параметров рига и объектов
- процедурное обновление трансформов, ссылок или вспомогательных объектов

Не надейся на `Python Tag` как на универсальный обработчик UI-команд. В C4D 2026 кнопки из `User Data` на `Null`-объекте могут не проксироваться в tag через `MSG_DESCRIPTION_COMMAND`.

Рекомендуемый базовый паттерн:
- состояние и анимируемые параметры хранятся в `User Data`
- командные действия живут в `GeDialog`
- `Python Tag` читает состояние и применяет его к объектам каждый кадр
- дефолты и структура `User Data` задаются декларативно, например через JSON

## 2. User Data в C4D 2026

### Что хорошо работает

Для большинства tool/rig/plugin-задач в `User Data` удобно хранить:
- `bool`
- `real`
- `link`
- группы `group`
- иногда `button`, но только если ты уверен в способе обработки

Типичный набор ролей:
- `real` для анимируемых численных параметров
- `bool` для включения режимов
- `link` для ссылок на объекты сцены
- `group` для структурирования UI

### Что лучше не смешивать

Не стоит делать `User Data` единственным местом для всего сразу:
- анимируемые параметры и runtime-state там уместны
- destructive actions и технические команды лучше выносить в `GeDialog`
- сложные многошаговые workflows лучше запускать через командный плагин или диалог

### Практический шаблон создания UD

Минимальный безопасный паттерн:

```python
bc = c4d.GetCustomDataTypeDefault(c4d.DTYPE_REAL)
bc[c4d.DESC_NAME] = "Orbit"
bc[c4d.DESC_MIN] = 0.0
bc[c4d.DESC_MAX] = 360.0
bc[c4d.DESC_STEP] = 0.1
bc[c4d.DESC_PARENTGROUP] = group_id
bc[c4d.DESC_CUSTOMGUI] = getattr(c4d, "CUSTOMGUI_REALSLIDER", None)
desc_id = obj.AddUserData(bc)
obj[desc_id] = 0.0
```

Полезные правила:
- задавай `DESC_NAME`, `DESC_MIN`, `DESC_MAX`, `DESC_STEP`
- группируй параметры через `DESC_PARENTGROUP`
- `DESC_CUSTOMGUI` ставь только если константа реально доступна
- значение по умолчанию присваивай после `AddUserData()`

### Почему лучше описывать UD декларативно

JSON-шаблон удобен, если нужно:
- централизовать структуру UI
- переиспользовать дефолты
- легче менять диапазоны и интерфейсы
- собирать разные конфигурации ригов

Хорошая схема шаблона:
- `groups`
- `params`
- `type`
- `name`
- `default`
- `min`
- `max`
- `step`
- `interface`
- `animatable`
- `link_ref`

Практика показывает, что отдельный шаблон для layout и централизованный словарь defaults сильно упрощают reset и обратную совместимость.

## 3. BaseContainer, DescID и чтение User Data

### Главное правило

`c4d.BaseContainer` не ведет себя как обычный Python `dict`.

Не делай так:

```python
value = bc.get(c4d.DESC_NAME)
```

Делай так:

```python
try:
    value = bc[c4d.DESC_NAME] or ""
except Exception:
    value = ""
```

### Почему это важно

В C4D 2026 это один из самых частых источников ложных ожиданий:
- `.get()` может отсутствовать
- часть полей контейнера может вести себя не так, как ожидается из Python-логики
- типы и метаданные иногда доступны не полностью или нестабильно

### Безопасное чтение всех User Data

Надежный runtime-подход:

```python
result = {}
for desc_id, bc in obj.GetUserDataContainer():
    try:
        name = bc[c4d.DESC_NAME] or ""
    except Exception:
        name = ""
    if not name:
        continue
    try:
        result[name] = obj[desc_id]
    except (AttributeError, TypeError):
        pass
```

Почему это хороший паттерн:
- за один проход собирается весь state
- нечитаемые элементы и группы просто пропускаются
- логика ниже работает уже с обычным Python-словарем
- код перестает зависеть от попыток дергать `BaseContainer` как `dict`

### Поиск UD по имени

Если нужно установить параметр по display-name:
- итерируйся по `GetUserDataContainer()`
- сравнивай `DESC_NAME`
- пропускай `DTYPE_GROUP`, если ищешь рабочий параметр

Это особенно важно, когда имя группы совпадает с именем одного из параметров.

## 4. UI: Attribute Manager vs GeDialog

### Когда достаточно Attribute Manager

`Attribute Manager` хорош для:
- слайдеров и чекбоксов
- object links
- параметров рига
- параметров, которые должен крутить аниматор на таймлайне

Если плагин по сути является rig/controller-системой, то `Attribute Manager` обычно должен быть основным UI для параметров состояния.

### Когда нужен GeDialog

`GeDialog` лучше подходит для:
- `Create Rig`
- `Create Motion Rig`
- `Reset`
- `Break User Data`
- import/export
- действия, меняющие сцену пачкой
- команды, требующие `Undo`, подтверждения или сообщений пользователю

### Важное ограничение C4D 2026

Практически подтвержденный кейс: `User Data`-кнопки на `Null`-объекте могут быть видимыми в `Attribute Manager`, но их нажатие не доходит до `Python Tag` как `MSG_DESCRIPTION_COMMAND`.

Вывод:
- не строй критичную командную логику на UD-кнопках, если объектом-носителем является `Null`
- команды reset/bake/break лучше выносить в `GeDialog`

### Видимость кнопок в Attribute Manager

Если кнопка создана, но не видна:
- проверь `DESC_HIDE`
- проверь, раскрыта ли секция `User Data`
- проверь, раскрыта ли нужная группа
- убедись, что выделен именно объект-носитель `User Data`

Но даже если кнопка отображается, это не означает, что ее message pipeline надежен.

## 5. Сообщения и обработка событий в 2026

### Description messages

В C4D 2026 данные description-сообщений стоит воспринимать как Python-структуру с ключом `"descid"`, а не как старый `BaseContainer`-объект с ожидаемым `"id"`.

Не закладывайся на:

```python
button_id = data.get("id")
```

Гораздо безопаснее:
- проверять наличие `"descid"`
- писать отдельный helper для извлечения `DescID`
- не смешивать legacy-ожидания разных версий API

### Python Tag не должен быть UI-router'ом

Если логика критична для production:
- `Python Tag` пусть занимается runtime-обновлением
- кнопочные команды переноси в командный плагин или `GeDialog`

Это уменьшает количество версионных сюрпризов.

## 6. Python API gotchas в C4D 2026

### `BaseContainer.get()` не использовать

Используй:
- `bc[id]`
- `try/except`
- промежуточный Python `dict`

### `AddCheckbox`

Сигнатура требует `initw` и `inith`.

Безопасный вызов:

```python
self.AddCheckbox(ID_SOME_CHECK, c4d.BFH_LEFT, 0, 0, "Option", initval=False)
```

### `AddComboBox`

Не рассчитывай на `customlayout` в Python API.

Безопасный вызов:

```python
self.AddComboBox(ID_COMBO, c4d.BFH_SCALEFIT, 200, 0)
```

### `MessageDialog`

Ожидает строку сообщения:

```python
c4d.gui.MessageDialog("Done")
```

### `ScrollGroupEnd`

Не во всех окружениях существует как метод. После `ScrollGroupBegin()` обычно нужно корректно закрывать layout через `GroupEnd()`.

### `SetTimer`

Сигнатуры могут отличаться между версиями. Перед использованием сверяйся с SDK именно своей версии C4D 2026 и не копируй вызов из старых примеров без проверки.

### `RegisterTagPlugin`

Сигнатуры и аргументы менялись между версиями. Если код пришел из старого примера:
- перепроверь `g`
- не полагайся на устаревшие keyword-аргументы
- тестируй регистрацию отдельно от остальной логики

### Градусы и радианы

Одна из самых частых логических ошибок:
- UI показывает градусы
- runtime еще раз конвертирует их как будто это уже не градусы
- в результате получается двойное преобразование

Практика:
- храни пользовательские углы в градусах
- переводи в радианы только в точке применения к rotation API
- внимательно проверяй, не навешан ли еще и `DESC_UNIT_DEGREE` вместе с дополнительным ручным преобразованием

### Custom GUI constants

Не все GUI-константы гарантированно доступны в каждом окружении.

Паттерн:

```python
gui_id = getattr(c4d, "CUSTOMGUI_RSSLIDER", None)
if gui_id is not None:
    bc[c4d.DESC_CUSTOMGUI] = gui_id
```

Это особенно актуально для:
- `CUSTOMGUI_REALSLIDER`
- `CUSTOMGUI_REALSLIDER_NOEDIT`
- `CUSTOMGUI_RSSLIDER`
- `CUSTOMGUI_BUTTON`

## 7. Runtime-архитектура плагина

### Рекомендуемая схема

Хорошо переносимая архитектура:

```text
User Data -> read once per frame -> normalize values -> apply to scene objects
```

Практические правила:
- читай все `User Data` за один проход
- работай дальше уже со словарем значений
- держи runtime-логику в отдельных функциях `_apply_*`
- не смешивай построение рига и логику тега в одном большом обработчике

### Обратная совместимость сцен

Если структура носителей параметров уже менялась между версиями плагина:
- сначала читай новый источник истины
- при отсутствии данных используй fallback на старую схему

Например:
- сначала читать `User Data` с корня рига
- потом, если данных нет, читать их со старого контроллера

Это простой и рабочий способ не ломать старые сцены.

### Object resolution

Не размазывай поиск нужных объектов по всему коду. Лучше:
- один resolver для объектов рига
- один resolver для motion-варианта
- ниже по коду работать уже с типизированной или хотя бы централизованной структурой

Это упрощает:
- reset
- break/bake
- optional features
- тестирование

### Optional features

Если плагин поддерживает расширения вроде inertia/lookahead/spring:
- подключай их только через явный `Enable`
- делай extension point, а не `if/elif` по всему основному коду
- не позволяй optional feature ломать базовый rig flow

### Осторожно с изменением иерархии во время expression execution

Изменение структуры объектов прямо во время работы expression/tag-логики может быть хрупким местом.

Риски:
- нестабильное поведение при undo/redo
- неожиданные side effects в сцене
- трудности в отладке

Если все же делаешь это:
- делай минимально
- проверяй, существует ли объект уже сейчас
- не перетасовывай иерархию без явной необходимости

## 8. Link-параметры и зависимости от окружения

`link`-поля удобны, но требуют явной стратегии fallback.

Полезный паттерн:
- если пользователь не указал внешний объект, подставить внутренний reference-object сцены
- не считать `None` ошибкой автоматически

Это особенно полезно для:
- target objects
- spline references
- focus helpers

## 9. Redshift, Motion Camera и edition-dependent API

Некоторые типы объектов и API доступны не везде.

Не рассчитывай без проверок на:
- `Redshift Camera`
- `Motion Camera tag`
- отдельные renderer-specific параметры

Правильный подход:
- сначала попытаться получить нужный type id или param id
- если объект/тег недоступен, использовать fallback
- если fallback невозможен, сообщить пользователю понятную причину

Для production-устойчивости:
- не падать с исключением только потому, что модуль отсутствует
- отделять optional functionality от core functionality

## 10. Undo, bake, reset и destructive actions

Все команды, которые меняют сцену заметно или необратимо, должны идти через явный command flow:
- найти активный объект или rig
- валидировать контекст
- начать `Undo`
- выполнить изменение
- завершить `Undo`
- вызвать `EventAdd()`
- показать понятный результат пользователю

Это относится к:
- `Reset`
- `Break User Data`
- bake параметров на реальные объекты
- созданию иерархий

### Reset-стратегия

Лучше хранить defaults централизованно:
- в `config`
- или в шаблоне `User Data`
- или в обоих местах, если один источник синхронизирует второй

Так reset не расползается по файлам и не начинает зависеть от случайных литералов.

### Break/Bake-стратегия

Если параметр перестает управляться `User Data` и должен стать обычной анимацией:
- перенеси анимированные значения на реальные параметры объектов
- только потом разрывай runtime-цепочку
- снимай служебные зависимости и теги явно

## 11. Debug checklist

Если что-то не работает, проверяй в таком порядке:

1. Параметр вообще создан в `User Data`.
2. Объект-носитель действительно выделен.
3. Нужная группа раскрыта в `Attribute Manager`.
4. Параметр читается через `GetUserDataContainer()`.
5. Имя параметра совпадает с тем, что ожидает runtime-код.
6. Нет ли конфликта имени между группой и полем.
7. Значение не падает на конвертации `float/bool/link`.
8. Тег висит на ожидаемом объекте.
9. Runtime-код берет данные из правильного объекта.
10. `EventAdd()` вызывается после действий, меняющих сцену.
11. `Undo` обернут правильно.
12. Не используется недоступный type id или renderer-specific API.

Дополнительно для command/UI-проблем:

1. Кнопка только видна или реально что-то вызывает.
2. Событие идет в `GeDialog.Command()` или ожидается ошибочно в `Python Tag`.
3. Сигнатура UI-метода подходит именно для C4D 2026.

## 12. QA checklist перед выпуском

Проверь хотя бы эти сценарии:

1. Создание плагином новой иерархии в пустой сцене.
2. Создание рига при выделенном `Null`.
3. Отображение всех `User Data` в `Attribute Manager`.
4. Анимацию численных параметров.
5. Работа `link`-полей с объектами сцены.
6. Поведение без назначенных ссылок.
7. Reset отдельных групп.
8. Reset всех параметров.
9. Bake или break workflow.
10. Открытие старой сцены после изменения структуры `User Data`.
11. Поведение без Redshift или без optional module.
12. Поведение в сцене с несколькими ригами.

## 13. Рекомендуемые паттерны

### Pattern: state in UD, commands in dialog

Хороший базовый split:
- `User Data` хранят состояние
- `GeDialog` выполняет команды
- `Python Tag` применяет состояние

### Pattern: read-once normalize-then-apply

Не дергай `obj[desc_id]` по всему коду много раз. Сначала собери все значения, затем:
- нормализуй типы
- подставь defaults
- вызови маленькие `_apply_*` функции

### Pattern: best effort compatibility

Для API, зависящего от версии или edition:
- использовать `getattr`
- держать fallback
- не считать отсутствие optional feature фатальной ошибкой

### Pattern: explicit source of truth

Явно определяй:
- где живет `User Data`
- кто читает параметры
- кто делает reset
- кто выполняет destructive commands

И документируй это в одном месте.

## 14. Антипаттерны

- считать `BaseContainer` полноценным словарем
- завязывать критичный reset на UD-кнопки в `Null` + `Python Tag`
- смешивать UI-команды и runtime-логику в одном обработчике
- делать двойную конвертацию градусов в радианы
- разбрасывать имена `User Data` строковыми литералами по всему проекту
- жестко полагаться на renderer-specific API без fallback
- менять иерархию в expression-коде без строгой необходимости

## 15. Минимальный blueprint нового плагина

Если делать новый plugin/rig/tool с нуля, безопасный стартовый blueprint такой:

1. В `config` собрать все имена, plugin ids, defaults и object names.
2. Отдельно описать структуру `User Data` в JSON или в одном builder-модуле.
3. Сделать helper для чтения всех `User Data` в `dict`.
4. Выделить runtime-функции `_apply_*`.
5. Команды `create/reset/break/import/export` вынести в `GeDialog` или command plugin.
6. Все edition-specific части завернуть в проверки доступности API.
7. Сразу продумать fallback для старых сцен, если структура может поменяться.

## 17. Главное, что стоит помнить

Если нужна короткая версия всей шпаргалки, то она такая:

- `User Data` хорошо подходят для анимируемого состояния.
- `GeDialog` лучше подходит для команд и reset-flow.
- `Python Tag` хорош для runtime-применения, но не как надежный UI-event router.
- `BaseContainer` нельзя бездумно использовать как обычный `dict`.
- C4D 2026 требует осторожности с message payload, сигнатурами UI-методов и edition-specific API.
- Делай fallback, явный source of truth и простой debug path.
